AURA — Architecture diagrams

Visual companion to the v1 build spec. Six diagrams covering every component and every hot path.

Diagram 1

System overview

Every component at rest, and the wires between them. Two Next.js apps under one hosting envelope, two Clerk instances, one shared Postgres reached under two roles. The dashed lines are outbound calls back to Twilio.

Key points
  • Customer app and admin app share the hosting envelope but are separate deployments, separate Clerk instances, separate Postgres roles.
  • The Postgres role determines tenant isolation. Not the URL, not the app code.
  • Inngest only connects to the DB for the nightly reconciliation job — never in the hot path.
Diagram 2

Inbound SMS flow

A patient texts back. Twilio hits the webhook, signature is verified, tenant is resolved by the receiving number, message is recorded, intent is checked, SSE pushes to the operator's open inbox.

Key points
  • Webhook routes bypass Clerk middleware. They authenticate by signature, not session.
  • Idempotency on twilio_sid — Twilio retries; the same SID never produces a duplicate row.
  • Webhook must return 200 in under 500ms. Heavy work goes to Inngest from inside the handler.
Diagram 3

Outbound SMS send flow

An operator replies from the inbox. Auth resolves tenant, the opt-out check sits between loading the patient and any Twilio call, and the status callback later updates the message's delivery state.

Key points
  • The opt-out check is between loading the patient and any Twilio call. Once Twilio is called the message is gone.
  • Status callback is asynchronous and may arrive seconds to minutes later — the schema handles the lag.
  • SSE event is fired locally, not from Twilio — so the sender's UI updates immediately, before delivery confirms.
Diagram 4

CSV upload flow

An operator drops a patient CSV. The route handler parses synchronously (fits in one request at the 5 MB / 10k row ceiling), dedupes by (tenant_id, phone), records per-row errors, and is idempotent on re-upload.

Key points
  • csv_imports row is created before parsing — every upload, including failures, leaves an audit trail.
  • Dedupe key is (tenant_id, phone) — two tenants can have the same patient phone without colliding.
  • Re-uploading is idempotent by construction. No client-side fingerprint required.
Diagram 5

Tenant provisioning flow

A new business signs up. Clerk hosts auth and org creation; webhooks back to the app provision the tenants row and users row. All writes are idempotent because webhooks retry and may arrive out of order.

Key points
  • One webhook endpoint, three event types. Signature is verified once at the top of the handler.
  • Clerk webhooks aren't ordered. Every write uses ON CONFLICT DO UPDATE — replays and reorderings produce the same final state.
  • Org creator gets role='owner'; invitees default to 'operator'. Promotion happens via Clerk's org UI.
Diagram 6

Super admin cross-tenant read flow

Internal staff view a tenant's data. The admin app opens the DB connection under app_admin, whose RLS policy is USING (true). Every cross-tenant read writes an admin_audit_log row in the same transaction.

Key points
  • The Postgres role swap is the isolation mechanism. App-code bugs don't break it; only role misconfiguration would.
  • Audit write is in the same transaction as the read. No path to "read succeeded, log failed."
  • Cross-tenant aggregations stay in SQL. Raw cross-tenant rows never enter app memory.
